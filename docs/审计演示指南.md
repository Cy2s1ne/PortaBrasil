# PortaBrasil 审计模块演示指南

这份指南用于向导师展示系统里的“AI 审计与财务复核”能力。演示重点不是只说有 AI，而是展示它能把业务数据、费用明细、成本记录和异常发现串成一个可追踪闭环。

## 1. 准备演示数据

本地 SQLite 演示环境：

```bash
cd Portabrasil-server
uv run python -c "from sql.database import get_database; get_database().initialize()"
sqlite3 instance/portabrasil.db < sql/demo_seed.sql
```

删除这批演示数据：

```bash
sqlite3 instance/portabrasil.db < sql/demo_seed_delete.sql
```

MySQL 演示环境：

```bash
cd Portabrasil-server
mysql -u root -p portabrasil < sql/schema.sql
mysql -u root -p portabrasil < sql/demo_seed.sql
```

删除这批演示数据：

```bash
mysql -u root -p portabrasil < sql/demo_seed_delete.sql
```

默认登录账号：

- `admin / admin123456`

演示数据里已经预置了 5 条清关业务、5 条流程记录、3 条成本记录、3 条审计运行、5 条审计发现，以及 2 条财务复核记录。

## 2. 推荐演示顺序

1. 登录系统，先展示首页、流程跟踪和成本分析页面，说明系统先沉淀了清关业务、费用和成本数据。
2. 打开数据库或接口，展示 `customs_business`、`customs_business_fee_item`、`customs_cost_record` 是审计输入。
3. 展示审计结果表：`ai_audit_run` 是一次审计运行，`ai_audit_finding` 是审计发现项。
4. 用正常案例 `S/2026-IMP-001` 对比异常案例 `S/2026-IMP-002`，说明系统能区分低风险和高风险。
5. 最后展示财务复核表：`ai_finance_review`、`ai_finance_item`，说明审计之后还能继续分析成本健康度。

## 3. 可直接展示的案例

正常案例：`S/2026-IMP-001`

- 风险等级：`LOW`
- 分数：`94`
- 讲法：借贷平衡、费用合计一致、成本记录可追踪，所以适合作为“通过审计”的基准案例。

异常案例：`S/2026-IMP-002`

- 风险等级：`HIGH`
- 分数：`62`
- 审计发现：
  - 借贷余额不一致，差额 `1,580.00 BRL`
  - 费用明细与业务总额存在 `1,600.00 BRL` 差异
  - 查验状态下仓储费用偏高，金额 `3,180.00 BRL`

中风险案例：`S/2026-IMP-004`

- 风险等级：`MEDIUM`
- 讲法：客户余额方向为 `FAVOR`，系统提醒财务确认退款、抵扣或后续账期处理。

## 4. 接口演示

先登录获取 token：

```bash
curl -X POST http://127.0.0.1:5001/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"admin123456"}'
```

查看审计运行列表：

```bash
curl http://127.0.0.1:5001/api/audit/runs \
  -H "Authorization: Bearer <access_token>"
```

查看某一次审计详情：

```bash
curl http://127.0.0.1:5001/api/audit/runs/1 \
  -H "Authorization: Bearer <access_token>"
```

手动触发一次业务审计：

```bash
curl -X POST http://127.0.0.1:5001/api/audit/business/1/run \
  -H "Authorization: Bearer <access_token>" \
  -H "Content-Type: application/json" \
  -d '{"cost_record_id":1}'
```

查看财务复核列表：

```bash
curl http://127.0.0.1:5001/api/finance/reviews \
  -H "Authorization: Bearer <access_token>"
```

## 5. 导师答辩讲法

可以这样介绍：

> 审计模块不是单独做一个静态页面，而是基于业务主表、费用明细和成本记录做规则/AI 复核。每次审计都会生成一条 `ai_audit_run`，记录风险等级、分数、检查项和输入快照；具体问题写入 `ai_audit_finding`，包括严重程度、证据、金额和处理建议。这样老师可以看到审计过程可追踪、结果可复查、异常可定位。

如果导师问“AI 在哪里”，可以回答：

> 后端 `app/services/audit_finance_service.py` 里封装了审计服务。当前可以通过 `LLM_PROVIDER=deepseek` 和 `DEEPSEEK_API_KEY` 切换到 DeepSeek 审计；配置 LLM API Key 时走模型审计，没有 API Key 或模型失败时自动走规则引擎，所以演示现场不会因为网络或密钥失败而中断。两种方式都会落到同一组审计表，保证结果可查询。

如果导师问“为什么要有审计表”，可以回答：

> 因为清关业务涉及税费、付款、退款和客户结算，单纯展示当前业务状态不够。审计表记录每次检查的时间、输入快照、风险等级和发现项，便于追责、复核和后续财务处理。

## 6. 数据库查询语句

审计运行总览：

```sql
SELECT id, business_id, cost_record_id, source, status, risk_level, score, summary, created_time
FROM ai_audit_run
ORDER BY created_time DESC;
```

审计发现项：

```sql
SELECT r.id AS run_id, b.s_ref, r.risk_level, f.severity, f.rule_code, f.title, f.amount
FROM ai_audit_run r
JOIN customs_business b ON b.id = r.business_id
LEFT JOIN ai_audit_finding f ON f.audit_run_id = r.id
ORDER BY r.id DESC, f.id ASC;
```

业务、成本、审计闭环：

```sql
SELECT
    b.s_ref,
    b.customer_name,
    c.record_no,
    r.risk_level,
    r.score,
    r.summary
FROM ai_audit_run r
JOIN customs_business b ON b.id = r.business_id
LEFT JOIN customs_cost_record c ON c.id = r.cost_record_id
ORDER BY r.created_time DESC;
```
